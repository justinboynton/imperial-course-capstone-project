# Datasheet — Capstone BBO Challenge Dataset

This datasheet follows the framework proposed by Gebru et al. (2021), *Datasheets for Datasets*, Communications of the ACM, 64(12), pp. 86–92. It documents the dataset used in the Stage 2 capstone challenge for Bayesian optimisation of black-box functions.

---

## 1. Motivation

### Purpose

The dataset exists to support a **constrained black-box optimisation (BBO) challenge** in which participants maximise eight unknown functions using a limited query budget. It serves as the empirical foundation for developing, evaluating and iterating Gaussian Process surrogate models and acquisition strategies under realistic conditions where each function evaluation is expensive and irreversible.

### Creators and funding

The initial data (seed observations) and the black-box evaluation oracle were created by Imperial College for the **Professional Certificate in Machine Learning and Artificial Intelligence** programme. The portal-submitted observations were generated by the participant (Justin Boynton) over a 10-week period. No external funding was involved; the dataset was produced as part of an assessed academic capstone project.

### Intended use

The dataset is intended for:

- Training and validating GP surrogate models for each of the eight functions
- Comparing acquisition strategies (UCB, EI, PI, Mean) under a constrained query budget
- Documenting the iterative decision-making process in sequential experimental design
- Supporting reflections on surrogate model selection, kernel choice and hyperparameter tuning

### What this dataset should NOT be used for

- **Benchmarking general-purpose optimisers.** The functions are proprietary to the course and their properties (smoothness, noise, dimensionality) are not representative of any standard benchmark suite (e.g., BBOB/COCO).
- **Training production ML systems.** Sample sizes (20–50 points per function) are far too small for supervised learning applications.
- **Drawing conclusions about specific real-world domains.** The function descriptions (contamination field, chemical yield, cake recipe, etc.) are pedagogical labels; the underlying equations are unknown and may not reflect actual domain physics.

## 2. Composition

### What do the instances represent?

Each instance is a **(input vector, scalar output)** pair representing a single evaluation of one of eight black-box functions. The input is a continuous vector in [0, 1]ᵈ (where d ranges from 2 to 8) and the output is a real-valued scalar.

### Sources of noise, error and redundancy

- **Stochastic output (F2).** F2 is explicitly noisy: repeated queries at nearby inputs can return different outputs. The heteroscedastic GP models this with per-point noise estimates derived from leave-one-out residuals.
- **Deterministic functions (F7 confirmed).** F7 was confirmed deterministic by submitting near-identical inputs in W2 and W5 and receiving outputs differing by < 0.002.
- **Zero-valued placeholders.** Portal submissions that have not yet received results are recorded as Y = 0.0 in `capstone_history.json`. These are not genuine observations and must be excluded from GP training. They appear as the W10 entries at the time of writing.
- **No duplicate instances.** Each portal submission used a unique input vector. Some are very close together (F2 W8–W9: distance 0.005) but none are exact duplicates.
- **Potential label noise in initial data.** The initial seed data was provided by the course facilitators. Its generation process is not documented; it may include measurement noise, rounding, or deliberate perturbation.

### Sensitive or confidential content

The dataset contains no personal, demographic, financial or health-related information. All instances are synthetic function evaluations.

## 3. Collection Process

### Initial seed data

The initial observations (`initial_inputs.npy`, `initial_outputs.npy`) were provided by the course facilitators as pre-computed NumPy arrays. The generation methodology is not disclosed — participants do not know whether the points were sampled uniformly, via Latin Hypercube, or by some other design. The sample sizes (10–40 per function) scale with dimensionality, suggesting a deliberate design-of-experiments approach.

### Portal submissions

Portal observations were collected one per function per week over 10 weeks (W1–W10), via the capstone submission portal. The process for each submission was:

1. Fit a GP surrogate on all available data (initial + prior submissions)
2. Evaluate an acquisition function (UCB, EI, PI or Mean) over 5,000 random candidates
3. Optionally apply manual hard-constraint clipping
4. Submit the input vector to the portal
5. Receive the scalar output later in the week (no same-week iteration)

Each submission is logged in `capstone_history.json` with full metadata: input vector, output, acquisition function, β, ξ, kernel type and timestamp.

### Ethical review and consent

No ethical review or consent process was required. The dataset is entirely synthetic — no human subjects, personal data or animal experiments are involved. The portal is operated by the course institution under its standard academic use policies.

### GDPR considerations

Not applicable. The dataset contains no personal data as defined under GDPR Article 4(1). The only identifying information in the repository is the participant's name and email in `README.md`, which is voluntarily published.

## 4. Pre-processing

### Cleaning steps applied

- **Y clipping.** Extreme output values are clipped to [−10¹², 10¹²] before GP fitting to prevent numerical instability in the kernel matrix.
- **NaN/Inf handling.** Any non-finite Y values are replaced with 0.0 before fitting (`np.nan_to_num`).
- **Zero-placeholder exclusion.** W10 entries with Y = 0.0 (results pending) are included in the JSON but excluded from GP training when detected.

### Y-transforms applied before GP fitting


| Transform               | Functions | Purpose                                                                                                                                       |
| ----------------------- | --------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `arcsinh(Y/s)`          | F1        | Spreads near-zero values across a wider range so the GP can distinguish magnitudes differing by dozens of orders                              |
| `standardize` (z-score) | F2–F8     | Zero-mean, unit-variance normalisation ensures acquisition parameters (β, ξ) have consistent meaning across functions with different Y scales |


These transforms are invertible and applied only during surrogate model fitting. All stored observations, plots and reported results use raw (untransformed) Y values.

### Labelling

No manual labelling was performed. Each instance's "label" is its scalar output, returned directly by the evaluation oracle.

### Raw data availability

The raw initial data is stored in `initial_data/function_{1..8}/` as `.npy` files. Portal submissions are stored verbatim in `capstone_history.json`. No data has been discarded, filtered or selectively excluded from the repository.

## 5. Uses

### Prior applications within this project

The dataset has been used for:

- **GP surrogate training and prediction** (`capstone_app.py`) — fitting per-function Gaussian Processes with function-specific kernels (Matérn 5/2, Matérn 3/2, Rational Quadratic, RBF) and acquisition functions
- **Kernel selection analysis** (`analysis/06_kernel_variants_ngboost.ipynb`) — LOO R² and 95% PI coverage comparison across kernel types
- **NGBoost evaluation** (same notebook) — testing tree-based probabilistic surrogates and rejecting them due to poor uncertainty calibration
- **F1 hotspot analysis** (`analysis/07_function1_hotspot_hunt.ipynb`) — model-free log-space analysis, radial decay estimation, sign boundary mapping
- **Neural network surrogate evaluation** (`analysis/05_nn_surrogate_analysis.ipynb`) — Deep Ensemble vs GP comparison at small n

### Inappropriate or prohibited uses

- **Reverse-engineering the black-box functions.** Attempting to reconstruct the underlying equations from the data violates the spirit of the challenge, which tests sequential decision-making under uncertainty rather than function identification.
- **Sharing with other participants.** Each participant's portal submissions are individual; sharing query results would compromise the assessment's integrity.
- **Using output values as ground truth for supervised learning.** The outputs are black-box evaluations with unknown noise characteristics, not verified labels. They should not be treated as a curated training set.

## 7. Maintenance

### Responsible party

Justin Boynton ([justin.boynton@gmail.com](mailto:justin.boynton@gmail.com)) is the sole maintainer of the portal submission data, analysis notebooks and surrogate model configurations.

The initial seed data and the evaluation oracle are maintained by the course facilitators. The participant has no ability to modify, update or correct the initial data or the oracle's behaviour.

### Update schedule

The dataset grows by exactly 8 observations per week (one per function) for the duration of the challenge (10 weeks, now complete). No further portal submissions will be added. The `capstone_history.json` file may be updated to correct metadata errors (e.g., missing acquisition settings for W1 submissions) but observation values (X, Y) are immutable once recorded by the portal.

### Error reporting

Any errors in the dataset (e.g., incorrectly transcribed portal results, misaligned X/Y arrays) should be reported via the repository's issue tracker or by contacting the maintainer directly. The `align_xy_pair()` function in `capstone_app.py` includes automated integrity checks that detect and correct length mismatches between X, Y and metadata arrays.

### Versioning

The dataset is versioned via Git. Tagged branches (`week-1`, etc.) provide snapshots at key milestones. The full commit history preserves every change to `capstone_history.json`.
